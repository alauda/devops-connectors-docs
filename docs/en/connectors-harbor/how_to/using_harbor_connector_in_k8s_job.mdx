# Building Images Using Harbor Connector in K8S Job

Before reading, please refer to [The General Logic of Using the Harbor Connector Reverse Proxy in K8S Workload](./using_harbor_connector_in_k8s.mdx)

## Feature Overview

This article will guide you on how to use the Harbor Connector to build images within a `Kubernetes Job` and push them to a Registry. With the capabilities of the Harbor Connector, ordinary users can operate without needing to touch or configure authentication credentials, thereby maximizing the security of these credentials.

When using the Harbor Connector within a Kubernetes Job, the following key points should be noted:

- Change the address of the target image being built to the connector's proxy address. [What is the Connector Proxy Address](../concepts/harbor_connectorclass.mdx#proxy_and_configuration_capabilities)
  - For example: harbor.example.cn/test/abc:v1 -> harbor.default.cluster.local/namespaces/harbor-connector-ns/connectors/harbor-connector-name/test/abc:v1
- Mount the configuration provided by the Connector.
- Configure `insecure registry` for the client tools.

This article will take `buildkitd` as an example and detail the process of creating a Kubernetes Job to complete the image building and pushing utilizing the `Harbor Connector`, without the need to configure authentication information on the client side.

## Prerequisites

- Create a Connector.
- A Containerfile for building the image.

### Connector

```shell
kubectl apply -f - <<EOF
apiVersion: connectors.alauda.io/v1alpha1
kind: Connector
metadata:
  name: harbor
spec:
  address: https://harbor.example.com # Replace with an accessible Harbor registry address
  auth:
    name: basicAuth
    secretRef:
      name: harbor-secret
  connectorClassName: harbor
---
apiVersion: v1
stringData: # Replace with actual authentication information
  password: username
  username: password
kind: Secret
metadata:
  name: harbor-secret
type: kubernetes.io/basic-auth
EOF
```

### Containerfile

To demonstrate the building and pushing process, we need to prepare a Containerfile. To simplify the operation, we will store the content of the Containerfile in a ConfigMap and mount it to the Pod using a Kubernetes Pod volume.

```shell
kubectl apply -f - <<EOF
apiVersion: v1
kind: ConfigMap
metadata:
  name: containerfile
data:
  Containerfile: |
    FROM scratch
    LABEL maintainer="example@example.com"
    WORKDIR /app
    ENV APP_VERSION="1.0.0"
EOF
```

## Operational Steps

### Creating the K8S Job with Forward Proxy

- Configure the forward proxy for the Harbor Connector, such as `http_proxy`, `https_proxy`, `no_proxy`, etc.

The `Harbor ConnectorClass` provides an out-of-the-box configuration that can be mounted through connector-csi.

```shell
cat << EOF | kubectl create -f -
kind: Pod
apiVersion: v1
metadata:
  generateName: buildkit-build-
spec:
  restartPolicy: Never
  containers:
  - name: buildkit
    image: moby/buildkit:v0.18.2
    securityContext:
      privileged: true
    env:
    - name: BUILDKITD_FLAGS
      value: "--config /etc/buildkit/buildkitd.toml"
    command:
    - /bin/sh
    - -c
    args:
    - |
      set -ex

      export http_proxy=$(cat /tmp/http.proxy)
      export https_proxy=$(cat /tmp/https.proxy)
      export HTTP_PROXY=$http_proxy
      export HTTPS_PROXY=$https_proxy
      export no_proxy=localhost,127.0.0.1
      export NO_PROXY=$no_proxy
      echo "Using proxy: http_proxy=$http_proxy, https_proxy=$https_proxy, no_proxy=$no_proxy"

      buildctl-daemonless.sh --debug \
      build \
      --progress=plain \
      --frontend=dockerfile.v0 \
      --opt filename=Containerfile \
      --local context=/workspace \
      --local dockerfile=/workspace \
      --output type=image,name=harbor.example.com/library/image:v1,push=true \
      --export-cache type=inline
    volumeMounts:
    - name: containerfile
      mountPath: /workspace
    - name: proxyconfig
      mountPath: /tmp/

  volumes:

  # Mounting the containerfile to the Pod
  - name: containerfile
    configMap:
      name: containerfile

  # Mounting proxyconfig to the Pod
  - name: proxyconfig
    csi:
      readOnly: true
      driver: connectors-csi
      volumeAttributes:
        connector.name: "harbor"
EOF
```

### Creating the K8S Job with Reverse Proxy

- To utilize the connector's proxy capabilities, we need to inject the authentication information required by the proxy into `config.json`. A volume must be provided to mount `config.json`.
- To allow the client to access the HTTP proxy, we need to configure `insecure-registries` for the client. A volume must be provided to mount `buildkitd.toml`.
- Modify the address of the pushed image to the connector's proxy address.

> For information on obtaining the Connector's proxy address, see: [Harbor Connector Class Proxy Information Description](../concepts/harbor_connectorclass.mdx#proxy_and_configuration_capabilities)

The Job content is as follows:

```shell
cat << EOF | kubectl create -f -
kind: Pod
apiVersion: v1
metadata:
  generateName: buildkit-build-
spec:
  restartPolicy: Never
  containers:
  - name: buildkit
    image: moby/buildkit:v0.18.2
    securityContext:
      privileged: true
    env:
    - name: BUILDKITD_FLAGS
      value: "--config /etc/buildkit/buildkitd.toml"
    command:
    - /bin/sh
    - -c
    args:
    - |
      set -ex
      buildctl-daemonless.sh --debug \
      build \
      --progress=plain \
      --frontend=dockerfile.v0 \
      --opt filename=Containerfile \
      --local context=/workspace \
      --local dockerfile=/workspace \
      --output type=image,name=c-harbor.default.svc.cluster.local/namespaces/harbor-connector-demo/connectors/harbor-connector/test-demo:v1,push=true \
      --export-cache type=inline
    volumeMounts:
    - name: containerfile
      mountPath: /workspace
    - name: registry-config
      mountPath: /root/.docker
    - name: buildkitd-config
      mountPath: /etc/buildkit

  volumes:

  # Mounting the containerfile to the Pod
  - name: containerfile
    configMap:
      name: containerfile

  # Mounting registry config.json to the Pod
  - name: registry-config
    csi:
      readOnly: true
      driver: connectors-csi
      volumeAttributes:
        connector.name: "harbor"
        configuration.names: "registry-config"

  # Mounting buildkitd.toml to the Pod
  - name: buildkitd-config
    csi:
      readOnly: true
      driver: connectors-csi
      volumeAttributes:
        connector.name: "harbor"
        configuration.names: "buildkitd"
EOF
```

## Running `harbor-cli` in a K8S Job via the Harbor Connector

Besides building images, you can also use the Harbor Connector to run the official `harbor-cli` inside a Kubernetes Job without placing the original Harbor username or password in the Pod. The Pod still receives the connector-generated Harbor CLI materials that `harbor-cli` needs at runtime, including `config.yaml`, `HARBOR_ENCRYPTION_KEY`, and the encrypted dummy `password` value stored in `config.yaml`. The Harbor ConnectorClass already renders a ready-to-use `harbor-cli-config`; see [Harbor CLI configuration](../concepts/harbor_connectorclass.mdx#harbor-cli-configuration) for the full list of mounted files.

Key points:

- Mount the `harbor-cli-config` configuration through `connectors-csi`; the mount provides `config.yaml`, `HARBOR_ENCRYPTION_KEY`, and a `.env` file that carries the forward-proxy environment variables. The generated `config.yaml` stores a dummy Harbor CLI `password` field in encrypted form.
- Before invoking `harbor-cli`, source the proxy `.env`, export `HARBOR_CLI_CONFIG`, and export `HARBOR_ENCRYPTION_KEY`.
- Always pass `-o json` (or `-o yaml`) to `harbor-cli` subcommands in a Job. `harbor-cli` defaults to an interactive TTY view and fails with `could not open a new TTY` inside a non-interactive container.

Example Job that lists a Harbor project and robot account through the Connector proxy:

```shell
cat << 'EOF' | kubectl create -f -
apiVersion: batch/v1
kind: Job
metadata:
  generateName: harbor-cli-
spec:
  backoffLimit: 0
  template:
    spec:
      restartPolicy: Never
      containers:
        - name: cli
          image: registry.goharbor.io/harbor-cli/harbor-cli@v0.0.18
          command: ["/bin/bash", "-c"]
          args:
            - |
              set -euo pipefail
              set -a; source /mnt/harbor-cli/.env; set +a
              export HARBOR_CLI_CONFIG=/mnt/harbor-cli/config.yaml
              export HARBOR_ENCRYPTION_KEY=$(cat /mnt/harbor-cli/HARBOR_ENCRYPTION_KEY)

              harbor-cli project list --name <your-harbor-project> -o json
              harbor-cli robot list -q 'name=<your-robot-account>' -o json
          volumeMounts:
            - name: harbor-cli
              mountPath: /mnt/harbor-cli
              readOnly: true
      volumes:
        - name: harbor-cli
          csi:
            readOnly: true
            driver: connectors-csi
            volumeAttributes:
              connector.name: "harbor"
              configuration.names: "harbor-cli-config"
EOF
```

Notes:

- The example uses the official `harbor-cli` image `registry.goharbor.io/harbor-cli/harbor-cli@v0.0.18`. You can replace it with any image that already contains a compatible `harbor-cli` binary.
- The Job Pod never sees the original Harbor password. It does receive `HARBOR_ENCRYPTION_KEY` and a generated `config.yaml` whose dummy `password` field is stored in Harbor CLI's encrypted format, because Harbor CLI needs both values to read the mounted configuration. That dummy value is not used as the real credential for Harbor access; real Harbor access is still authorized through the Connector proxy loaded from `.env`.
- For multi-Connector setups, change `connector.name` to `connectors: "<conn-a>,<conn-b>"` and switch contexts with `harbor-cli context switch <namespace>/<connector>`. This requires the `enable-multi-connector` feature flag. For details, see [Feature Flags](../../overview/feature_flags.mdx).

## Operational Results

You can check whether the Job has executed successfully with the following command:

```shell
$ kubectl get job
```

## Summary

We have completed the entire process of "Building Images Using Harbor Connector in K8S Job." During Job execution, the workload does not need the original Harbor username or password. Depending on the integration mode, the Pod receives connector-generated proxy credentials or Harbor CLI runtime materials such as `HARBOR_ENCRYPTION_KEY` and the encrypted Harbor CLI config with a dummy `password` value, while real Harbor access is still mediated by the Connector. This reduces direct exposure of the original Harbor credential in client workloads.
