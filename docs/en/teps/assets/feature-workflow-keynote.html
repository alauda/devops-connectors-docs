02·WHERE WE STOP TODAY
The Problem

The workflow ends at code merged. The product question doesn’t.

No traceability.

OpenSpec change → operator bundle → QA result → regression status → production incident: there is no thread that ties them together.

No coordination.

A cross-repo feature lives across connectors, extensions, operator, and plugin. Nobody can answer “where is it” without reading four histories.

No DoD enforcement.

The real definition of done — regression green on a built bundle, with no production hotfix for 14 days — is tracked in chat. Drivers re-discover it every release.

03·PROPOSAL
One layer up
A feature-level workflow above per-repo OpenSpec changes.
01

One driver.

A single person walks the feature from start to ship. Pause and handoff are first-class; concurrent drivers are not.

02

One command family.

/feature:* — one command per stage. Step-by-step, no skill lookup required.

03

One umbrella per feature.

A folder under docs/en/design/ with state, design, QA packet, outcomes, and post-ship observations.

04

One maturity signal.

Every intervention classified by blocker category. The team sees where to invest — no single percentage to game.

06·STORIES ARE THE UNIT, REPOS ARE THE FAN-OUT
Key idea · 02

A four-repo feature is not four independent designs — it’s one story that fans out.

Story 1 · backend · p0

Shared OAuth-app credential model

  • connectors design-change
  • extensions design-change
  • operator (docs) mechanical-followup
Story 4 · backend · p0 · deps: 1

ConnectorsGit / GitLab CRD extensions

  • operator design-change

Story group = one story + N per-repo OpenSpec changes linked to it. Per-story review replaces per-repo-per-story review: one signature for the whole group.

design-change — full OpenSpec pre-apply cycle (proposal → specs → design → tasks → bdd-design) and an independent reviewer signature.

mechanical-followup — only tasks.md + PR. Content is fully determined by a parent design-change. No independent design cycle. Inherits review.

WHY  → The connectors-operator sync-install-manifests PR that follows a connectors annotation change is mechanical. Classing it as such saves one reviewer handoff and one full OpenSpec cycle per feature.
07·DISPATCH BY DEPENDENCY, NOT BY REPO
Key idea · 03

The driver writes the graph once. Dispatch, rebase, unblock — automatic.

/feature:implement reads dependencies.md and dispatches stories in order. Blocked stories stay as draft PRs with a banner; auto-rebase and re-dispatch happen when blockers merge — no human polling.

Story mutation is first-class. Realise mid-build that story 6 is really two (GitHub + GitLab variants)? /feature:story --split 6 runs a fast design-review on the two new stories and keeps the feature moving.

DRILL-DOWN
/feature:tracks — story-grouped board
/feature:dispatch <repo> — scope to one
08·DESIGN REVIEW IS A REAL GATE
Key idea · 04

Pivots are normal. Two in a row → the workflow offers a POC.

  • approved

    Advances to plan. For sensitive features, requires a security-labeled reviewer.

  • pivot

    Direction change. Returns to design. Second consecutive pivot → workflow offers POC.

  • rework

    Direction OK, details wrong. Same loop-back, smaller correction.

  • poc

    Throwaway branch spike. Outcome feeds the next /feature:design pass. Excluded from maturity metrics.

09·PAUSE AND HANDOFF ARE DIFFERENT OPERATIONS
Key idea · 05

Stepping off for a week is not the same as changing who owns it.

Pause

/feature:pause [--reason=…]
  • Same driver. You’ll pick it back up.
  • Sub-agents stay alive. Draft PRs stay open; banner comment added to each.
  • No rebuild. /feature:resume picks up exactly where it was.
  • Jira comment notes the pause.
  • Only /feature:resume, status, handoff, cancel work while paused.

Handoff

/feature:handoff <new-driver> [--reset]
  • New driver. Ownership transfers.
  • Sub-agents NOT cancelled by default. New driver inherits in-flight draft PRs.
  • --reset is opt-in when the new driver wants a clean slate.
  • handoff.md snapshot: open questions, deferred decisions, context.
  • New driver’s first command is just /feature:status.
COMPOSE  → a paused feature can be handed off (new driver inherits paused state). An active feature can be handed off without pausing (new driver takes over mid-flight).  |  NOT SUPPORTED  → concurrent drivers. At any moment one driver; Jira handles multi-person via separate stories.
10·SECURITY, PROPORTIONAL TO RISK
Key idea · 06

Risk is computed, not guessed. The overlay does the rest.

Risk · low
low

  • No security artifacts
  • CI scans as today
Risk · standard
standard
  • Most features land here
  • Security section in design-review.md
  • CI scans enforced
Risk · sensitive
sensitive
  • Auth · secrets · TLS · RBAC · network egress
  • Standalone threat-model.md
  • Security-labeled reviewer at design-review
  • /feature:security-sign-off gate before ship
TRIGGER CHECKLIST AT INIT  →  handles credentials?  ·  touches TLS/CA?  ·  modifies RBAC or admission?  ·  changes egress to third party?  ·  new exposed endpoint?  ·  cluster-scoped resources?
RESULT  →  any “yes” in the sensitive list raises risk to sensitive; otherwise standard. Overrides are logged and surfaced at every gate.

12·RELEASE REALITY
Key idea · 08

Customers can’t test the week we ship. A 14-day window was aspiration.

2026.04 monthly
2026.05 monthly
2026.06 monthly
2026.07 LTS · every 4th
2026.08 monthly
2026.09 monthly

Monthly non-LTS releases for most customers; LTS every 4th. Production feedback on anything non-trivial takes weeks to months — longer than any single-feature umbrella should stay open.

14-day landed window

Ends before most customers touch the feature. Either catches nothing useful, or forces every umbrella to linger for months. The feature umbrella is the wrong container for this kind of feedback.

Archive at ship · feedback on the epic

Feature umbrella closes cleanly at ship. The Jira epic — long-lived by nature — is the right container for post-release bugs, tech debt discovered later, and new stories that spawn from customer feedback.

13·TWO UMBRELLAS, ONE WORKFLOW
Key idea · 09

The Jira epic is long-lived. The shippable slice is bounded.

Epic umbrella · persistent

docs/en/design/epics/<jira>-<slug>/

One per Jira epic. Holds cross-story artifacts and survives many releases.

  • research.md — cross-story findings
  • design-overview.md — architectural shape
  • stories.md — story list with priority
  • dependencies.md — story-level edges
  • post-release-log.md — bugs, tech debt, new stories
  • shipped-features/ — back-links to archived features
/feature:epic-init  ·  story-start  ·  bug-link  ·  epic-status  ·  epic-close
Feature umbrella · bounded

docs/en/design/<feature-jira>-<slug>/

One per shippable slice. Inherits from the epic when story-started. Runs the feature pipeline (slide 4) and archives at ship.

  • feature.md, state.yaml, handoff.md
  • design + tech-design + test design
  • qa-packet / acceptance / regression
  • maturity-report.md
  • retrospective.md (runs before ship)
/feature:init (standalone)  ·  /feature:story-start <id> (from epic)
ONE-STORY EPIC  → No separate epic umbrella needed. /feature:init is the entrypoint; the feature is the epic.  |  MULTI-STORY EPIC  → /feature:epic-init creates the long-lived container; each story gets its own feature umbrella via /feature:story-start.

16·DELIVERY
How we ship this

Seven incremental phases. Each is its own OpenSpec change.

Scaffolding

  • this TEP merged
  • state schema committed (epic + feature)
  • /feature:init (standalone feature)
  • /feature:status · /feature:next
  • /feature:handoff · /feature:pause · /feature:resume
  • /feature:state-repair (audited)
  • --risk checklist · collision scan · WIP signal

Epic umbrella

  • /feature:epic-init (long-lived container)
  • /feature:story-start <id> (branch feature)
  • /feature:epic-status (board across releases)
  • /feature:bug-link (post-release)
  • /feature:epic-close (archive)
  • post-release-log.md append path

Design flow

  • /feature:research (full only · inherits from epic)
  • /feature:design (goal + breakdown + deps)
  • /feature:design-review (pivot / rework / POC)
  • /feature:poc (optional spike)
  • /feature:story (add / split / merge / defer)

Implement fan-out

  • /feature:plan (story groups · design-change vs mechanical)
  • /feature:implement (dispatch by dependencies.md)
  • /feature:tracks (story-grouped board)
  • /feature:dispatch <repo>

Post-merge

  • /feature:integrate (Tekton observe)
  • /feature:qa (executes test design)
  • /feature:accept ∥ /feature:docs
  • /feature:regress

Retro · ship · archive

  • /feature:security-sign-off (sensitive only)
  • /feature:retro (opt-out for light)
  • /feature:ship → immediate archive
  • maturity-report.md
  • shipped-features/<feature>.link on the epic

Maturity tracking

  • per-command blocker classification
  • /feature:maturity (this feature)

17·WHAT THIS IS NOT
Scope guardrails

Just as important: what we are not building.

✗

Not a Jira replacement.

Jira writes are comments and one state transition — nothing more. The integration is small enough to swap.

✗

Not a new CI system.

The Tekton pipelines stay. We observe them at integrate; we don’t drive them.

✗

Not a forced migration.

Existing OpenSpec changes keep working. New features opt in at /feature:init.

✗

Not a 90% automation chase.

Maturity replaces rate. Judgment-only stages are the honest floor, not a miss.

✗

Not a gate that blocks small fixes.

light profile shrinks artifacts but every validation gate still runs. Retro is opt-out for trivial fixes.

✗

Not concurrent drivers.

One driver at a time. Pause and handoff are first-class. Multi-person coordination is a Jira problem.
