- 2026-05-21 · DEVOPS-43952 · (process) — Driver feedback from feature-flow optimisation meeting: `/feature:poc` is currently an opt-in stage **between** `design-review` and `plan`, gating implementation on a separate validation pass. In practice, teams want the POC to run **inside** the same stage as `design` — i.e., `/feature:design` produces docs **and** runs the throwaway-branch validation against the real upstream system before the docs are considered "done" and shipped to `design-review`. Proposal: introduce a sub-step `design-validate` (or fold poc.md production into `/feature:design` when the driver opts in) so design+POC complete as one atomic stage. [open]
- 2026-05-21 · DEVOPS-43952 · (process) — Driver decision: a single "implementation" task / stage should bundle **all four** convergence pillars rather than spreading them across separate stages: (1) code + (2) automated tests (BDD), (3) multi-round independent-agent reviews, and (4) manual verification on a real environment when feasible. Goal is to exit the implementation stage with the implementation already in a "reliable state" rather than handing partially-verified code to a downstream QA / accept stage. Current `/feature:implement` only mandates code + tests; reviews are a separate `requesting-code-review` skill invocation and manual verification has no clear home before `/feature:qa`. Proposal: extend `/feature:implement` exit criteria to require (a) BDD green in CI, (b) ≥1 round of `code-review-subagent` per repo with disposition recorded, and (c) when a live dev environment exists for the upstream system, manual-verification evidence captured as a short note under the story's PR description. Same "POC-in-design" rationale: catch real-system breakage in the stage that owns the change, not 2-3 stages later. [open]
