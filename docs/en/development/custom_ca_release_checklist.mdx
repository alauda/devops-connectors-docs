# Custom CA Certificates Release Checklist

## Scope

Release-gate manual QA checklist for the custom CA certificates feature (introduced in v1.10.0 under the `enable-custom-ca-certs` feature flag). Run this checklist once per release-candidate bundle on a production-shaped TLS demo cluster. It is the human-eye safety net for integration-surface and real-cluster quirks that the unit tests (Layer 1), BDD scenarios (Layer 2), and structural checks (Layer 3 — extension plugin.yaml audit and RBAC golden-file test) do not cover. It does not replace those automated layers. Budget roughly 45 minutes of hands-on time per bundle.

## Environment prerequisites

- Demo cluster with the TLS toolchain deployed via `demo-env-tls.yaml` (GitLab, Harbor, Nexus, and SonarQube all serving HTTPS with a private CA).
- Feature flag `enable-custom-ca-certs=true` on `ConfigMap connectors-system/connectors-config`; controller, proxy, and API deployments rolled after the flip.
- Global CA Secret in `connectors-system` labeled `connectors.cpaas.io/ca-cert=true` with the demo-toolchain root CA in a `.crt` / `.pem` key.
- At least one per-connector CA case — a Connector with `spec.caCertSecretRef` pointing to a Secret in the same namespace.
- Pre-existing QA fixtures available (fresh-cluster reset: recreate `qa-cacert-flag-on/qa-invalid-pem`, `qa-cacert-flag-on/qa-missing-secret`, `qa-cacert-flag-on/qa-valid-cacert`, `testing-connector-cacert-e2e-wrong` nginx + RoleBinding, `kube-public/maven-public`, `orbitmall/qa-oci-harbor`, `orbitmall/orbitmall-harbor`).
- `kubectl` context pointed at the demo cluster with cluster-admin; `jira` CLI available for posting the release-gate report.

## Known classes of failure that only show manually

- **Generated-manifest gaps (missing env vars on Deployments — as in DEVOPS-43811).** Look for plugin Deployments missing `CONFIG_NAME`, `POD_NAMESPACE`, or other feature-flag-loading env vars. Most likely to surface when a plugin's `CACertReady=True` but `AuthReady=False` with `x509: certificate signed by unknown authority` — a sign the plugin never read the feature flag and silently fell back to `InsecureSkipVerify=false` against an unknown CA.
- **Generated-RBAC gaps (missing kubebuilder markers — as in DEVOPS-43812).** Look for controller log lines like `events is forbidden: User "system:serviceaccount:connectors-system:..." cannot create resource "events"`. Most likely to surface when the `CACertReady` condition is correctly `False` on a Connector but `kubectl describe connector` shows an empty Events list — the reconciler called `Eventf()` but the API rejected it with 403.
- **Protocol-level / multi-pod gaps where one process in the chain lacks context about the Connector (as in DEVOPS-43824).** Look for split signals: `CACertReady=True` on the Connector (controller loaded the per-connector Secret) but `AuthReady=False` on the same Connector with an `x509` error (the plugin pod on the other side of the RPC never received the per-connector CA). Most likely to surface when a Connector relies *exclusively* on `spec.caCertSecretRef` with no global labeled Secret available to compensate; pass-through works if a global CA Secret happens to sign the same tool.
