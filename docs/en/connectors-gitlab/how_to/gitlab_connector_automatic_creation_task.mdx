# GitLab Connector Automatic Creation Tekton Task

This page is the parameter, workspace, and result reference for the
`gitlab-connector-automatic-creation` Tekton Task. For the operator
walkthrough — Pattern A, Pattern B, rotate, recreate, expired-fall-through,
and the operations runbook — see
[Auto-Create GitLab Tenant Group and Connector with Tekton](./using_gitlab_connector_automatic_creation_task.mdx).
For the underlying model — how the admin Connector is consumed through
`gitlabconfig`, the three-path GAT lifecycle, and the tenant-side
surface — see
[GitLab CLI Configuration and Auto-Creation Patterns](../concepts/glab_cli_config.mdx).

## Installation \{#installation}

Set the variables, then run the command below:

- `REGISTRY`: the registry host that mirrors `registry.alauda.cn:60070` (the catalog `glab` and `kubectl` tool images live under this prefix).
- `TARGET_NS`: the namespace where you want to install the Task. The catalog tool-image ConfigMaps that the Task references are already installed in `kube-public` by the connectors-operator.

        # ensure-group.sh -- step 1 of gitlab-connector-automatic-creation/0.1
        #

        # path via `glab api groups`. Honours both deployment patterns:

        #   Pattern A: tenantGroup is a top-level path (no slash). Admin
        #              identity is a user PAT with `can_create_group`.
        #   Pattern B: tenantGroup is `umbrella/tenant`. Admin identity is the
        #              umbrella's GAT with owner access.
        #
        # Inputs (env): TENANT_GROUP, VERBOSE, GLAB_CONFIG_DIR (CSI mount).
        # Outputs    : prints group id + path; writes the group id to
        #              /workspace/state/tenant-group-id (next step reads it).
        #
        # This script never echoes the admin PAT/GAT. `set +x` is the default;
        # `verbose=true` env enables tracing on non-secret paths only.

        set +o xtrace 2>/dev/null || true

        : "${TENANT_GROUP:?ensure-group: TENANT_GROUP is required}"
        : "${VERBOSE:=false}"
        : "${GLAB_CONFIG_DIR:=/workspace/gitlabconfig}"
        : "${STATE_DIR:=/workspace/state}"

        mkdir -p "${STATE_DIR}"

        # Logs follow Harbor's harbor-connector-automatic-creation style:
        # sentence-case narrative ("Created tenant group ..."), no per-call API
        # hints, no bash xtrace. ERROR lines carry full context to stderr.

        # Run a glab/curl command up to 3 times if its combined output looks like
        # a transient DNS resolution failure. Reason: in CI's kind cluster the
        # in-cluster coredns occasionally returns NXDOMAIN/SERVFAIL for valid
        # hostnames during heavy back-to-back BDD runs; live GitLab instances can
        # also exhibit transient DNS in production.
        with_dns_retry() {
          local out rc attempt=1
          while :; do
            set +e
            out="$("$@" 2>&1)"
            rc=$?
            set -e
            if [[ ${rc} -eq 0 ]] || [[ ${attempt} -ge 3 ]]; then break; fi
            case "${out}" in
              *"no such host"*|*"server misbehaving"*|*"i/o timeout"*|*"dial tcp: lookup"*|*"Could not resolve host"*|*"Temporary failure in name resolution"*)
                echo "WARN: transient DNS failure (attempt ${attempt}/3), retrying in 3s" >&2
                sleep 3
                attempt=$((attempt + 1))
                continue
                ;;
            esac
            break
          done
          printf '%s' "${out}"
          return ${rc}
        }

        # Bootstrap glab credentials silently from the CSI mount. The mount is
        # read-only with mode 0444 (CSI default); glab itself rejects any
        # config.yml that isn't 0600 ("failed to read configuration"). Copy the
        # config into a writable XDG home and re-point GLAB_CONFIG_DIR (which
        # glab honours natively as the config-dir override) at that writable
        # copy so glab no longer reaches into the read-only mount.
        prepare_glab() {
          local glab_home="${HOME:-/tmp}/.config/glab-cli"
          mkdir -p "${glab_home}"
          if [[ ! -f "${GLAB_CONFIG_DIR}/config.yml" ]]; then
            echo "ERROR: glab config not found at ${GLAB_CONFIG_DIR}/config.yml -- is the gitlabconfig CSI mount bound?" >&2
            exit 1
          fi
          cp "${GLAB_CONFIG_DIR}/config.yml" "${glab_home}/config.yml"
          chmod 600 "${glab_home}/config.yml"
          export GLAB_CONFIG_DIR="${glab_home}"

          # glab v1.82+ resolves the target host from --hostname / GITLAB_HOST
          # / the current git remote / config-default, in that order. Our config
          # has exactly one host entry, but glab does NOT auto-pick it -- it
          # falls through to gitlab.com by default, then 401s because the
          # gitlab.com PAT we don't have. Pull the host + token out of the
          # mounted config and export them so glab's env-var fallback works.
          GITLAB_HOST=$(grep -E '^\s{2}[A-Za-z0-9._-]+:\s*$' "${glab_home}/config.yml" \
                         | head -1 | tr -d ': ' )
          GITLAB_TOKEN=$(awk '/^\s{4}token:/ { print $2; exit }' "${glab_home}/config.yml")
          if [[ -z "${GITLAB_HOST}" || -z "${GITLAB_TOKEN}" ]]; then
            echo "ERROR: could not parse host/token from ${glab_home}/config.yml" >&2
            exit 1
          fi
          export GITLAB_HOST GITLAB_TOKEN
        }

        # URL-encode a path so `umbrella/tenant` survives `GET /groups/:id`.
        url_encode_path() {
          local p="$1"
          printf '%s' "${p}" | sed 's|/|%2F|g'
        }

        # Read api_protocol from the glab config so direct-curl calls (used for
        # endpoints that glab encodes incompatibly, e.g. POST /members) target
        # the correct scheme.
        read_glab_protocol() {
          local cfg="${GLAB_CONFIG_DIR}/config.yml"
          GLAB_PROTOCOL="$(awk -F'[: ]+' '/^[[:space:]]*api_protocol:[[:space:]]/{print $3; exit}' "${cfg}")"
          if [[ -z "${GLAB_PROTOCOL}" ]]; then
            GLAB_PROTOCOL="https"
          fi
        }

        # Detect the admin token's user identity and whether it's a Group
        # Access Token (Pattern B umbrella GAT) vs a regular user PAT.
        #
        # GitLab's `/api/v4/personal_access_tokens/self` returns the active
        # token's metadata (works for PATs and GATs alike); the `user_id` field
        # points at the underlying user record. Group bots have `bot: true` and
        # a `username` of the shape `group_<id>_bot_<suffix>`.
        #
        # Sets globals on success:
        #   ADMIN_USER_ID         numeric id of the user the token belongs to
        #   ADMIN_IS_GAT_BOT      true|false
        #   ADMIN_BOT_USER_ID     same as ADMIN_USER_ID when ADMIN_IS_GAT_BOT=true; else empty
        detect_admin_identity() {
          read_glab_protocol
          local self_url user_url self_resp self_code user_resp user_code
          self_url="${GLAB_PROTOCOL}://${GITLAB_HOST}/api/v4/personal_access_tokens/self"
          set +x
          self_code="$(with_dns_retry curl -skS -o /tmp/identity_self -w '%{http_code}' \
            -H "PRIVATE-TOKEN: ${GITLAB_TOKEN}" "${self_url}" || true)"
          self_resp="$(cat /tmp/identity_self 2>/dev/null || true)"
          rm -f /tmp/identity_self
          ADMIN_USER_ID=""
          ADMIN_IS_GAT_BOT=false
          ADMIN_BOT_USER_ID=""
          if [[ "${self_code}" != "200" ]]; then
            # Older GitLab releases (<14.5) lack /self -- ownership verification
            # is skipped (Pattern A then proceeds without conflict-detection
            # against pre-existing groups owned by other users).
            return 0
          fi
          local user_id
          user_id="$(printf '%s' "${self_resp}" | jq -r '.user_id // empty')"
          if [[ -z "${user_id}" ]]; then
            return 0
          fi
          ADMIN_USER_ID="${user_id}"

          user_url="${GLAB_PROTOCOL}://${GITLAB_HOST}/api/v4/users/${user_id}"
          user_code="$(with_dns_retry curl -skS -o /tmp/identity_user -w '%{http_code}' \
            -H "PRIVATE-TOKEN: ${GITLAB_TOKEN}" "${user_url}" || true)"
          user_resp="$(cat /tmp/identity_user 2>/dev/null || true)"
          rm -f /tmp/identity_user
          if [[ "${user_code}" != "200" ]]; then
            return 0
          fi

          local is_bot username
          is_bot="$(printf '%s' "${user_resp}" | jq -r '.bot // false')"
          username="$(printf '%s' "${user_resp}" | jq -r '.username // empty')"
          if [[ "${is_bot}" == "true" ]] && [[ "${username}" =~ ^group_[0-9]+_bot_ ]]; then
            ADMIN_IS_GAT_BOT=true
            ADMIN_BOT_USER_ID="${user_id}"
          fi
        }

        # When the tenant group already exists, verify the admin actually has
        # Owner access (direct OR inherited) before reusing it. Without this
        # check, an unrelated user's group at the same path would be silently
        # adopted -- catastrophic for the connector contract.
        verify_admin_ownership() {
          local group_id="$1"
          if [[ -z "${ADMIN_USER_ID:-}" ]]; then
            # /personal_access_tokens/self unavailable -- can't verify. Log and
            # let the script proceed (older GitLab releases).
            echo "WARN: Admin user id unknown; skipping ownership verification on group ${group_id}" >&2
            return 0
          fi
          set +x
          local url resp http_code
          url="${GLAB_PROTOCOL}://${GITLAB_HOST}/api/v4/groups/${group_id}/members/all/${ADMIN_USER_ID}"
          http_code="$(with_dns_retry curl -skS -o /tmp/own_resp -w '%{http_code}' \
            -H "PRIVATE-TOKEN: ${GITLAB_TOKEN}" "${url}" || true)"
          resp="$(cat /tmp/own_resp 2>/dev/null || true)"
          rm -f /tmp/own_resp
          if [[ "${http_code}" == "404" ]]; then
            echo "ERROR: group path conflict; existing owner does not match admin (admin user_id=${ADMIN_USER_ID} is not a member of the existing group at ${TENANT_GROUP})" >&2
            exit 1
          fi
          if [[ "${http_code}" != "200" ]]; then
            echo "ERROR: failed to verify admin ownership of group ${group_id}: HTTP ${http_code} ${resp}" >&2
            exit 1
          fi
          local access_level
          access_level="$(printf '%s' "${resp}" | jq -r '.access_level // 0')"
          if [[ "${access_level}" -lt 50 ]]; then
            echo "ERROR: group path conflict; existing owner does not match admin (admin has access_level=${access_level}, requires Owner=50)" >&2
            exit 1
          fi
        }

        # NOTE: an earlier attempt added an `ensure_bot_owner_membership`
        # helper to add the umbrella GAT's bot user as a *direct* Owner of
        # freshly-created tenant subgroups (so that ensure-gat could mint a
        # subgroup GAT). GitLab rejects this with
        #   HTTP 400 {"message":{"user_id":["project bots cannot be added to other groups / projects"]}}
        # Group share (`POST /groups/<sub>/share`) granted access but did NOT
        # unblock the GAT-mint endpoint either. Pattern B's design needs to
        # change before TC2/TC9 can be automated -- see the FIXME blocks on
        # those scenarios in tektoncd.feature.

        # Look up the group by full path. Returns the JSON object on stdout, or
        # empty string + non-zero on 404.
        lookup_group() {
          local path_enc; path_enc="$(url_encode_path "$1")"
          with_dns_retry glab api "groups/${path_enc}" 2>/dev/null || return 1
        }

        # Create a new group. Pattern A: no parent_id. Pattern B: parent_id of
        # the umbrella, looked up by parent path.
        #
        # A "path has already been taken" rejection means a group already
        # exists at this path, owned by someone we can't see (typically a
        # different user's *private* group that lookup_group couldn't return).
        # That is a path conflict we MUST surface clearly -- silently
        # proceeding would mean adopting an unrelated org's tenant group.
        create_group() {
          local full_path="$1"
          local name="${full_path##*/}"
          local parent_path="${full_path%/*}"
          local out rc

          set +e
          if [[ "${parent_path}" == "${full_path}" ]]; then
            # No slash -- top-level group (Pattern A).
            out="$(with_dns_retry glab api -X POST groups -f "name=${name}" -f "path=${name}" -f "visibility=private")"
            rc=$?
          else
            # Subgroup -- need the parent's id (Pattern B).
            set -e

          local curl_rc

            exit 1

              echo "       Recommend revoking unused tokens at this tenant group." >&2

              echo "ERROR: GitLab rejected token expiry; check the instance-wide max_token_expiry policy: HTTP ${http_code} ${resp}" >&2
              exit 1
            fi
            echo "ERROR: GitLab refused to mint GAT: HTTP ${http_code} ${resp}" >&2
            exit 1
          fi
          local id token
          id="$(printf '%s' "${resp}" | jq -r '.id')"

          token="$(printf '%s' "${resp}" | jq -r '.token')"

## Description

Use `glab` and `kubectl` to create or reuse a tenant GitLab group plus
optional subgroups, mint a Group Access Token, and write the cluster-side
tenant GitLab Connector + auth Secret in one TaskRun. Supports Pattern A
(top-level group + `can_create_group` user PAT) and Pattern B (subgroup
under a pre-existing umbrella + admin user PAT with Owner on the
umbrella). Token refresh follows a three-path lifecycle: rotate,
recreate, expired-fall-through.

## Steps

The Task runs four sequential steps. All steps share an in-memory
`emptyDir` volume mounted at `/workspace/state` for non-secret state and
a tmpfs (`emptyDir.medium=Memory`) volume mounted at `/workspace/secrets`
for the GAT. The tmpfs volume is removed on Task exit.

| Step | Image | What it does |
| ---- | ----- | ------------ |
| `ensure-group` | `gitlabCliImage` | Look up `tenantGroup` via `glab api groups/:path`. If absent, create it (Pattern A: top-level; Pattern B: with `parent_id` resolved from the umbrella). Writes the group id and full path to `/workspace/state/`. |
| `ensure-subgroups` | `gitlabCliImage` | Iterate `subgroups[]` and create each idempotently as a child of the tenant group. A 409 from `glab api` is treated as success. Writes a JSON array of materialised subgroup paths. |
| `ensure-gat` | `gitlabCliImage` | List existing GATs at the tenant group, find the one whose name matches this Task's prefix, and pick one of three paths: rotate-in-place, recreate (identity drift or expired-fall-through), or create fresh. Writes the new GAT id to `/workspace/state/access-token-id` and the secret value to `/workspace/secrets/token`. |
| `apply-kubernetes-resources` | `kubectlImage` | Server-side apply the tenant Connector and Secret into the connector namespace. The Connector's `spec.address` comes from `connector_address` in the `gitlabconfig` mount. The Secret is `type: connectors.cpaas.io/gitlab-pat-auth` with `stringData.token` set from `/workspace/secrets/token`. Honours the optional `kubeconfig` workspace for cross-cluster apply. |

The Task `stepTemplate` enforces a non-root, capability-dropped security
context (`runAsUser: 65532`, `runAsGroup: 65532`,
`allowPrivilegeEscalation: false`, all capabilities dropped). The tool
images need to be runnable as a non-root user.

## Three-Path GAT Lifecycle Summary

| Existing GAT state | Identity match? | Action |
| ------------------ | --------------- | ------ |
| None | n/a | **Create** a fresh GAT and write the Secret. |
| Healthy and identity matches | yes | **Rotate in place** via `POST /groups/:id/access_tokens/:token_id/rotate`. Old token invalidated atomically; Secret updated in place. |
| Healthy but identity differs (`scopes`, `accessLevel`, or `subgroups[]` changed) | no | **Recreate**: revoke the existing GAT, mint a fresh one with the requested identity, rewrite the Secret. |
| Expired (`expires_at < now`) | n/a | **Expired-fall-through**: best-effort revoke, mint a fresh GAT, rewrite the Secret. The TaskRun reports `Succeeded`. No manual operator intervention required. |

See
[Three-Path GAT Lifecycle](../concepts/glab_cli_config.mdx#three-path-gat-lifecycle)
for the model and
[Step-by-Step — Expired-Fall-Through (Self-Healing)](./using_gitlab_connector_automatic_creation_task.mdx#step-by-step-expired-fall-through-self-healing)
for the recovery walkthrough.

## Security Notes

- The admin GitLab credential never enters the Pod environment. The
  `gitlabconfig` mount carries a short-lived proxy token; the real admin
  PAT/GAT lives behind the Connectors proxy.
- The tenant GAT is written to a tmpfs volume (`emptyDir.medium=Memory`)
  with mode 0600 and removed on Task exit.
- Shell tracing (`set -x`) is **never** active around code paths that
  read the GAT, regardless of `verbose=true`.
- The `secret` parameter is a bare resource name in the connector
  namespace; values containing `/` are rejected so a typo cannot land
  the GAT in a different namespace.
- `apply-kubernetes-resources` calls `kubectl auth can-i create secrets
  -n <connector-namespace>` before applying anything, surfacing missing
  RBAC and typos in `connector=<ns>/<name>` together.
